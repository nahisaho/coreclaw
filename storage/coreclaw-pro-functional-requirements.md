# Core Claw Pro 機能要件定義書

本書は、現行のCoreClawオープン版コードベースを踏まえ、製品版として追加で必要となる機能要件を整理したものである。既存の実験管理、チャット、スキル、Marketplace、GitHub連携、外部チャネル連携の上に、エディション制御とライセンス管理を追加することを目的とする。

## 1. 文書の目的

- Core Clawを製品として提供するために不足している機能要件を定義する。
- Core Claw OpenとCore Claw Proの機能境界を明確化する。
- ライセンス生成、付与、検証、失効、制限適用の仕様を明文化する。
- 今後の実装、UI設計、テスト設計、運用設計の基礎とする。

## 2. 前提

- 現行CoreClawには、実験管理、My Skills連携、GitHub認証、外部チャネルグループ管理などの基盤機能が存在する。
- 現行CoreClawには、製品ライセンス、契約状態、利用プラン、機能制限を扱う恒久的な仕組みは存在しない。
- Core Claw Openはライセンス未付与でも利用可能な無償版とし、Core Claw Proは有効ライセンスにより制限解除される有償版とする。

## 3. エディション定義

| Edition | 概要 |
|---|---|
| Core Claw Open | ライセンス未付与時に利用できる制限付き版 |
| Core Claw Pro | 有効なライセンスが付与された場合に利用できるフル機能版 |

## 4. 要件対象範囲

本書では、とくに以下を追加要件の対象とする。

- ライセンス管理
- エディション判定
- 機能制限の強制適用
- ライセンス状態のUI表示
- 期限切れと失効時の挙動
- ライセンス関連の監査と運用

## 5. 機能要件

### 5.1 ライセンス情報管理

| ID | 要件 | 内容 |
|---|---|---|
| PFR-01 | ライセンス情報保持 | システムは、ライセンスキー、対象 GitHub アカウント名、申込日、有効期限、エディション種別、状態を保持できなければならない。 |
| PFR-02 | ライセンス状態管理 | システムは、未登録、有効、期限間近、失効、無効の状態を判定し保持できなければならない。 |
| PFR-03 | ライセンス登録 | システムは、利用者が UI からライセンスキーを登録できなければならない。 |
| PFR-04 | ライセンス更新 | システムは、既存ライセンスを新しいキーで更新できなければならない。 |
| PFR-05 | ライセンス削除 | システムは、登録済みライセンスを削除し、Open 相当へ戻せなければならない。 |
| PFR-06 | 永続化 | システムは、ライセンス情報を再起動後も復元できるよう永続化しなければならない。 |

### 5.2 ライセンス生成と発行

| ID | 要件 | 内容 |
|---|---|---|
| PFR-07 | 発行入力項目 | システムは、ライセンス発行時に少なくとも GitHub アカウント名と申込日を受け付けられなければならない。 |
| PFR-08 | ライセンス生成規則 | システムは、GitHub アカウント名、申込日、エディション種別をもとに一意なライセンスキーを生成できなければならない。 |
| PFR-09 | 有効期限算出 | システムは、申込日を起点として 1 年後を有効期限として算出しなければならない。 |
| PFR-10 | 改ざん検知 | システムは、ライセンスキー文字列の改ざんを検知できなければならない。 |
| PFR-11 | 再発行 | システムは、同一 GitHub アカウントに対する再発行と更新履歴を区別して管理できなければならない。 |
| PFR-12 | 発行記録 | システムは、ライセンス発行日、対象者、期限、状態変更履歴を記録できなければならない。 |

### 5.3 ライセンス検証

| ID | 要件 | 内容 |
|---|---|---|
| PFR-13 | 起動時検証 | システムは、起動時に登録済みライセンスの整合性と有効期限を検証しなければならない。 |
| PFR-14 | 設定保存時検証 | システムは、ライセンス登録または更新時にキー形式と対象情報の整合性を検証しなければならない。 |
| PFR-15 | GitHub アカウント照合 | システムは、設定済み GitHub ユーザー名または認証済みアカウント名と、ライセンス対象の GitHub アカウント名を照合できなければならない。 |
| PFR-16 | 期限切れ判定 | システムは、現在日時が有効期限を超えた場合にライセンスを失効として扱わなければならない。 |
| PFR-17 | 利用前検証 | システムは、Pro 限定機能の実行前に有効ライセンスを再確認しなければならない。 |
| PFR-18 | 判定結果取得 API | システムは、現在のエディション、ライセンス状態、制限内容を返す API を提供しなければならない。 |

### 5.4 エディション制御

| ID | 要件 | 内容 |
|---|---|---|
| PFR-19 | Open 既定動作 | システムは、有効ライセンスが存在しない場合、Core Claw Open として動作しなければならない。 |
| PFR-20 | Pro 切替 | システムは、有効ライセンスが確認できた場合、Core Claw Pro として動作しなければならない。 |
| PFR-21 | 即時反映 | システムは、ライセンス登録、更新、削除、失効を行った際、再起動なしまたは明示的な再読み込みでエディション状態を反映できなければならない。 |
| PFR-22 | サーバー側 enforcement | システムは、UI 上の非表示や無効化だけでなく、API と実行制御の両方でエディション制限を強制しなければならない。 |

### 5.5 Open / Pro 機能制限

| ID | 要件 | 内容 |
|---|---|---|
| PFR-23 | My Skills 制限 | システムは、Core Claw Open では My Skills の利用を禁止しなければならない。 |
| PFR-24 | My Skills 解放 | システムは、有効な Pro ライセンスがある場合にのみ My Skills の設定、import、update、sync を許可しなければならない。 |
| PFR-25 | ChatGroup 上限制限 | システムは、Core Claw Open では作成または登録できる ChatGroup 数を 2 つまでに制限しなければならない。 |
| PFR-26 | ChatGroup 上限超過防止 | システムは、Open で上限に達している場合、新規 ChatGroup 作成または登録要求を拒否しなければならない。 |
| PFR-27 | Pro の上限緩和 | システムは、Core Claw Pro では ChatGroup 数の制限を解除、または別途定義された上限値まで緩和できなければならない。 |
| PFR-28 | 失効後制限再適用 | システムは、Pro ライセンスが失効した場合、My Skills を利用不可に戻し、ChatGroup 上限を Open の制限へ戻さなければならない。 |
| PFR-29 | 既存資産保全 | システムは、失効後も既存データを削除してはならず、Open 制限を超える ChatGroup や設定は読み取り専用または縮退状態で保持しなければならない。 |

### 5.6 期限切れ・猶予・更新案内

| ID | 要件 | 内容 |
|---|---|---|
| PFR-30 | 期限接近通知 | システムは、ライセンス期限の 30 日前、7 日前、1 日前を目安に利用者へ更新案内を表示できなければならない。 |
| PFR-31 | 失効通知 | システムは、ライセンス失効時に Open へ切り替わったことと制限内容を明示しなければならない。 |
| PFR-32 | 更新導線 | システムは、期限切れまたは期限接近時にライセンス更新画面または案内情報へ遷移できる導線を提供しなければならない。 |

### 5.7 UI / 画面要件

| ID | 要件 | 内容 |
|---|---|---|
| PFR-33 | ライセンス設定画面 | システムは、Settings 内または専用画面でライセンスの登録、更新、削除、状態確認を行える UI を提供しなければならない。 |
| PFR-34 | 状態表示 | システムは、現在の Edition、ライセンス状態、有効期限、対象 GitHub アカウントを画面上で確認できるようにしなければならない。 |
| PFR-35 | 制限理由提示 | システムは、Open 制限により利用できない機能に対して、無効化理由と Pro で解放される旨を表示しなければならない。 |
| PFR-36 | ChatGroup 上限表示 | システムは、Open で利用中の ChatGroup 数と上限値を視覚的に確認できるようにしなければならない。 |
| PFR-37 | My Skills 表示制御 | システムは、Open の場合に My Skills 関連操作を無効化し、Pro ライセンス登録導線を表示しなければならない。 |

### 5.8 API / 実行制御要件

| ID | 要件 | 内容 |
|---|---|---|
| PFR-38 | ライセンス状態取得 API | システムは、フロントエンドが現在のライセンス状態と制限内容を取得できる API を提供しなければならない。 |
| PFR-39 | ライセンス登録 API | システムは、ライセンスキーの登録と検証を行う API を提供しなければならない。 |
| PFR-40 | 機能ゲート共通化 | システムは、My Skills、ChatGroup 作成、将来の Pro 限定機能で共通利用できる機能ゲート判定部品を提供しなければならない。 |
| PFR-41 | エラー応答標準化 | システムは、ライセンス不足や上限超過時に、利用者向け説明が可能な標準エラーコードとメッセージを返さなければならない。 |

### 5.9 監査とサポート運用

| ID | 要件 | 内容 |
|---|---|---|
| PFR-42 | ライセンス操作監査 | システムは、ライセンス登録、更新、削除、検証失敗、失効判定を監査ログとして記録できなければならない。 |
| PFR-43 | 問い合わせ支援 | システムは、サポート対応のためにライセンス状態、対象 GitHub アカウント、失効日、直近検証結果を参照できなければならない。 |
| PFR-44 | 障害切り分け | システムは、ライセンス不一致、期限切れ、GitHub アカウント不一致、改ざん検知を区別して記録・表示できなければならない。 |

## 6. データ要件

- ライセンス情報は設定情報と分離し、少なくともキー本体、対象GitHubアカウント、申込日、有効期限、Edition、状態、最終検証日時を保持すること。
- ライセンスキーの平文保存可否は別途設計判断とするが、少なくとも改ざん検知に必要な検証情報を保存できること。
- 監査ログは、実行者、実行日時、操作内容、結果を追跡できること。

## 7. 業務ルール

- ライセンスの有効期間は申込日から1年間とする。
- ライセンス対象はGitHubアカウント単位とする。
- Core Claw Openはライセンス未登録でも継続利用可能とする。
- Core Claw OpenではMy Skillsを利用不可とする。
- Core Claw OpenではChatGroup数を2つまでとする。
- Core Claw Proは有効ライセンス期間中のみ利用可能とする。
- 失効後はOpenに自動的に縮退させる。

## 8. 今後の拡張候補

- 複数ライセンス種別への拡張
- 法人契約向けの席数管理
- オフライン検証トークンまたはオンライン検証API
- ライセンス更新購入導線との外部決済連携
- 機能別アドオンライセンス

## 9. 実装優先度

### 優先度 A

- ライセンス登録、保存、検証
- Open/Pro判定
- My Skills制限
- ChatGroup2件上限制限
- 期限切れでのOpen縮退
- ライセンス状態表示

### 優先度 B

- 期限接近通知
- 監査ログ
- 標準エラーコード
- 共通機能ゲート

### 優先度 C

- 再発行履歴管理
- サポート向け参照ビュー
- 将来の複数プラン対応

## 10. 補足

本書は、現行CoreClawの実装済み機能に対して、製品化のため追加で必要となる要件を定義したものである。既存の機能要件定義書が現行機能の整理であるのに対し、本書はCore Claw Proの提供条件と制限制御を仕様化する役割を持つ。
